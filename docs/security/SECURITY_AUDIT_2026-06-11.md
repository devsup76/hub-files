# Security Audit — woahh Payment Platform

**Date:** 2026-06-11
**Author:** Lead security engineer (remediation owner)
**Scope:** Multi-tenant storefront + payment state machine (Stripe + Square) + RLS isolation + edge functions + secrets/ops posture
**Status of findings:** All VERIFIED in code/migrations. De-duplicated across audit dimensions.

---

## 1. Executive Summary

woahh has a **fundamentally sound security architecture** that is currently undermined by **two classes of acute, exploitable failures**: (a) leaked live credentials sitting in the dev environment and chat history, and (b) a migration-ordering regression that re-opened a previously-closed anon data leak. The money path itself is well-designed — cards are tokenized (no PCI card storage), totals are server-validated (C1), per-org Square OAuth replaces the old global token, and SECURITY DEFINER RPCs mask secrets. But several of those defenses are **staged-not-applied, UI-only, or silently reverted**, so the *design* is safer than the *live system*.

**Overall security-posture rating: C+ / "At Risk — strong bones, urgent gaps."** Architecture would rate B+/A-; live-state credential exposure and the storefront-RPC regression drag the effective posture down to C+. This is **fixable in days, not months** — most CRITICALs are rotate-a-key or ship-one-migration.

**The single most dangerous risk:** the **live GitHub `ghp_` PAT in `.git/config` + the pasted `sbp_` Supabase admin token**. Either one grants an attacker the ability to ship arbitrary code to production (Cloudflare rebuilds `main`) or read every merchant's plaintext Square OAuth tokens and the service-role key. Everything else is downstream of "the keys are out." **Rotate first, fix code second.**

Five-bullet summary:

- **Credentials are exposed and must be rotated NOW.** A live `ghp_` GitHub PAT is in `.git/config` (write access → ship code to prod), the `sbp_` Supabase admin token was pasted into chat (reads every merchant's *plaintext* Square access/refresh tokens + the service-role key), and the Vault auto-decline/Square-refresh service-role-key copy must rotate in lockstep. No secret scanning, no CI, no pre-commit hooks exist to catch the next leak.
- **A go-live migration silently re-opened a closed anon leak.** `20260611000700` (publish-gate) re-created `get_public_storefront` with the OLD 7-column denylist, dropping ~13 masks — re-exposing every published merchant's Square merchant identity, payment-readiness flags, `contact_email`, unsalted account-recovery question/answer hashes, and financial counters to any anonymous visitor. This is an account-takeover chain, not just disclosure.
- **Several documented defenses are not actually live.** The org-payment-column write guard (`20260609050000`) is "staged for founder review," `online_card_enabled` is a UI-only gate the charge functions ignore, customers can self-edit their own loyalty balance over REST, and an injected `initial_status` lets a guest drop an unpaid order straight into the kitchen — bypassing the manual-capture/owner-confirm hold.
- **Cross-tenant money integrity has real gaps.** The Square webhook trusts a customer-visible `reference_id` (order UUID) with no org binding (a connected merchant can flip a rival's order to "paid"), and the `sandbox-direct` global-token affordance resurrects the exact BLK-2 fund-misrouting the per-org OAuth model was built to kill.
- **Abuse/DoS surfaces are wide open and self-amplifying.** Guest checkout and public reservations have no captcha/rate-limit (unbounded bot orders, bookings that lock every table, email-reputation burn), and order placement decrements stock *before* payment with **nothing restocking on decline/abandon** — a trivial inventory-DoS that "sells out" limited items in seconds.

## 3. Detailed Findings

### F1 — CRITICAL — Live GitHub `ghp_` PAT in git remote URL
**Location:** `/workspaces/GrowthHub/repo/.git/config` — `[remote "origin"] url = https://ghp_CA0q...@github.com/devsup76/business-growth-hub.git`
**Risk:** A live classic `ghp_` PAT for the `devsup76` account is stored in plaintext and used for every push. Classic tokens are coarse-scoped; this one has at least `repo` write. Cloudflare Pages rebuilds production `woahh.app` from `main`, so write access = ship arbitrary code to the live payment platform.
**Exploit:** Anyone who reads `.git/config` (or sees `git remote -v` in a log/transcript/screen-share) clones with the token, commits a one-line change to `square-payment/index.ts` that exfiltrates each merchant's resolved Square access token (or sets `application_fee` to themselves), and pushes to `main` → production rebuild. They own the money path for every merchant, or pivot to any repo `devsup76` can write.
**Fix (quick):** (1) Revoke the PAT now. (2) `git remote set-url origin https://github.com/devsup76/business-growth-hub.git` and auth via gh CLI / SSH deploy key — never an inline-URL token. (3) Review the GitHub audit log + recent `main` commits for unauthorized pushes. (4) Enable branch protection + required review on `main`. (5) Re-issue as fine-grained, single-repo, short-expiry.

### F2 — CRITICAL — Plaintext Square OAuth tokens, readable by the leaked `sbp_` admin token
**Location:** `migrations/20260610010000_square_connections.sql:34-35` (`access_token`/`refresh_token` as `text`; RLS on, zero policies → service-role-only); written raw by `square-oauth-connect/index.ts:199-200`.
**Risk:** Every merchant's Square access + refresh tokens are stored as plaintext `text`. The only protection is RLS-deny-by-default (service-role read). The leaked `sbp_` admin token (pasted into chat, used repeatedly → treat as in the wild) grants full project access incl. service-role DB reads, defeating that design entirely. Tokens carry `PAYMENTS_WRITE`/`PAYMENTS_WRITE_IN_PERSON`/`ORDERS_WRITE`/`PAYMENTS_READ` against each merchant's real Square account.
**Exploit:** Attacker with the `sbp_` token runs `SELECT org_id, access_token, refresh_token FROM square_connections`, then calls Square `CreatePayment`/`RefundPayment` directly against each merchant's account — drain refunds to themselves, issue charges, or pull full transaction history — entirely outside woahh's logging. Same admin token also reaches the Stripe secret, Resend, ClickSend, Anthropic, VAPID, and the service-role key.
**Fix (medium):** (1) Rotate the service-role key and revoke the `sbp_` admin token now; re-issue scoped/short-lived. (2) After rotation, force a Square re-OAuth (stored tokens are compromised) and revoke old grants in Square. (3) Encrypt `access_token`/`refresh_token` at rest with Supabase Vault (`vault.create_secret` per connection) or app-layer envelope encryption keyed by a KMS secret **not** reachable with the same admin token. (4) Alert on any direct `SELECT` of `square_connections` outside the known edge functions.

### F3 — CRITICAL — Publish-gate migration silently reverted the `get_public_storefront` mask
**Location:** `migrations/20260611000700_gate_public_storefront_on_publish.sql:27-64` (`CREATE OR REPLACE get_public_storefront`) — sorts LAST, so it is the authoritative live body, overriding the full denylists in `20260609060000` (H-5) and `20260610070000`. Anon caller: `src/services/api.ts:215`. Recovery chain: `account-recover/index.ts:28-29,131-132,144-148`.
**Risk:** `get_public_storefront` is the anon SECURITY DEFINER RPC every published storefront loads; it `RETURNS public.organizations` (full row) and relies on a hardcoded `r.col := NULL` denylist. The publish-gate migration re-created the function with only the ORIGINAL 7 masks (`owner_phone`, `owner_full_name`, `abn`, `business_address`, `stripe_account_id`, `phone_otp_hash`, `phone_otp_expires_at`), silently dropping ~13: `security_questions`, `contact_email`, `square_merchant_id`/`square_location_id`/`square_payment_ready`, `charges_enabled`, `payouts_enabled`, the email/SMS usage+topup counters, `total_donations_cents`, `founding_merchant`, and `settings`. No later migration re-masks it. `organizations.security_questions` is `jsonb [{question, answer_hash}]` of **unsalted** SHA-256 answer hashes; `account-recover` verifies `SHA-256(answer.trim().toLowerCase())` against exactly those, then fires `resetPasswordForEmail` → owner takeover. The publish gate makes it *worse* by scoping the leak to PUBLISHED (real, money-taking) merchants.
**Exploit:** Anon, no auth: `customerSupabase.rpc('get_public_storefront', {p_slug:'<published-slug>'})` (slugs are enumerable via `/eat`). Response now contains `security_questions` (questions + unsalted hashes), `contact_email`, `charges_enabled`/`payouts_enabled`/`square_payment_ready`, Square ids, and per-org revenue counters. Crack the small-answer-space hashes offline (pet/street/school), then `POST account-recover {action:'verify', email, answers:[...]}` → `resetPasswordForEmail` → full account takeover (orders, payouts, refunds, customer PII). Even without takeover, every merchant's payment posture + private counters leak to anonymous visitors. Verified the frontend reads none of these fields off this RPC, so re-masking breaks nothing.
**Fix (quick to ship, medium to do right):** Ship a new highest-numbered migration that `CREATE OR REPLACE`s `get_public_storefront` merging the publish-gate JOIN with the COMPLETE denylist from `20260610070000`. **Better (the IMP-1 fix):** replace `RETURNS public.organizations` with an explicit ALLOWLIST projection (only `id`, `name`, `subdomain_slug`, `business_type`, `logo_url`, marketplace display fields, `payment_provider`, and a `settings` subset). Salt the recovery-answer hashes. Add a regression test asserting the anon response never contains `security_questions`/`contact_email`/`square_*`/counters, plus a CI check that the live function definition matches the highest-numbered migration.

### F4 — HIGH — Client-controlled `initial_status` injects unpaid orders into the kitchen
**Location:** `migrations/20260608020000_c1_server_side_order_total.sql:268` (`v_status := COALESCE(NULLIF(p_payload->>'initial_status',''),'pending')`); set client-side at `RestaurantStorefront.tsx:639`; backstop gap at `20260609040000:124` (auto_decline only reaps `awaiting_confirmation`); capture toggle at `stripe-payment-intent/index.ts:116`.
**Risk:** The payment-safety model assumes a card-required online order is created at `awaiting_confirmation` so it sits unpaid out of the kitchen *and* uses MANUAL capture. That status is chosen only in client code; the RPC inserts whatever `initial_status` the payload carries. The `v_is_org_member` trust gate governs only the price floor, not the status. The RPC is GRANTed to anon+authenticated. `auto_decline_stale_orders` filters `status = 'awaiting_confirmation'`, so an injected `pending` order is never reaped, and `stripe-payment-intent` sets `automatic` capture for any non-`awaiting` status — the manual-capture invariant is lost.
**Exploit:** Anonymous guest mints an anon session (normal flow), calls `create_order_with_inventory` with real line_items, a valid `p_total ≥ floor`, and `initial_status='pending'` (or `'completed'`). Skip the card dialog. The order appears as a normal ticket the kitchen cooks; `payment_status` stays `unpaid`; no auto-decline fires. Repeat for free meals at scale.
**Fix (quick):** Make `initial_status` server-authoritative for untrusted callers: when NOT `v_is_org_member`, force `v_status := 'awaiting_confirmation'` (ignore client value); only org members may pass other statuses. Backstop: extend `auto_decline_stale_orders` (or a companion sweep) to reap stale `pending`/`preparing` orders with `payment_status='unpaid'` past the timeout.

### F5 — HIGH — Square webhook trusts `reference_id` (victim order UUID) with no org binding
**Location:** `square-webhook/index.ts:165-186` (`setOrderPayment`), `:122-133` (dispatch); `reference_id` = victim order UUID set at `square-payment/index.ts:347`.
**Risk:** Webhook signatures use a SINGLE per-application HMAC key shared by every connected merchant. `setOrderPayment` updates `orders.payment_status` keyed on `square_payment_id` OR, as a fallback, `reference_id = order UUID`, with NO check that the payment's merchant/location belongs to the order's org. The no-downgrade guard excludes `(paid,refunded,partially_refunded,canceled)` but NOT `unpaid`/`authorized`, so `unpaid→paid` is reachable cross-tenant. Order UUIDs are discoverable via `/order/{id}` links + receipts.
**Exploit:** Attacker completes normal woahh+Square onboarding (own Square account on the shared app). They learn a victim order UUID, `CreatePayment` in their OWN account with `reference_id=<victim UUID>`, let it COMPLETE. Square sends an app-signed `payment.updated`; the webhook matches via the `reference_id` fallback and sets the victim's order to `paid` though no money hit the victim's account. The victim ships goods for an order never charged.
**Fix (medium):** Read the Square `merchant_id` from the event envelope, load the matching `square_connections` row, and verify the resolved order's `organization_id` maps to that connection before mutating money state. Match only by `square_payment_id` issued by THIS org; never fall back to the customer-visible `reference_id` without an org/merchant cross-check. Reject events whose `merchant_id` has no connection. (Folds in F31's single-key concern.)

### F6 — HIGH — Customers can self-edit their own loyalty point balance
**Location:** `migrations/20260419061246:48-53` (policy "Customers update own record" on `public.customers`); columns `total_points`/`milestone_spend_cents` at lines 17-18.
**Risk:** The self-UPDATE policy is `FOR UPDATE TO authenticated USING (user_id = auth.uid()) WITH CHECK (user_id = auth.uid())` with NO column restriction. RLS can't do column-level `WITH CHECK`, and no later migration overrides it or adds a loyalty-pinning trigger. The legitimate write path is the separate `adjust_loyalty_points` RPC (`Loyalty.tsx:65-66` comments "clients can no longer write total_points directly"). `total_points` is redeemed for rewards. So the customer has full direct write control over their own redeemable balance + `marketing_opt_in`/opt-out flags/`saved_addresses`. (Note: `adjust_loyalty_points` is referenced by the frontend but absent from any repo migration — schema drift; the guarded path exists only as an RPC while the table stays directly writable.)
**Exploit:** A logged-in customer calls the REST API: `PATCH /rest/v1/customers?user_id=eq.<self> {"total_points":9999999,"milestone_spend_cents":9999999}`. RLS allows it, bypassing the staff-validated RPC. They redeem free rewards never earned. Repeatable across every merchant org they have a row in.
**Fix (quick):** Add a BEFORE UPDATE trigger on `public.customers` (mirroring `guard_org_payment_columns`): when `auth.uid()` is non-null (untrusted end-user write), pin `NEW.total_points`/`NEW.milestone_spend_cents`/consent timestamps back to OLD; SECURITY DEFINER/service writes (`auth.uid()` NULL) pass through. Alternatively REVOKE UPDATE on those columns and route all mutation through the RPC.

### F7 — HIGH — Owner can self-activate payment-routing columns on `organizations`
**Location:** `migrations/20260418045819:31-32` (policy "Owners update their org" — USING only); mitigation `20260609050000_guard_org_payment_columns.sql` (header: "STAGE FOR FOUNDER REVIEW before applying").
**Risk:** `"Owners update their org" FOR UPDATE USING (owner_id = auth.uid())` has no `WITH CHECK`, so Postgres reuses USING — re-asserts ownership but does not restrict columns. `payment_provider`, `square_payment_ready`, `square_merchant_id`, `square_location_id`, `charges_enabled`, `payouts_enabled`, `stripe_account_id` are all owner-writable over REST. The only mitigation is `guard_org_payment_columns`, whose own header documents it as staged-not-applied — so on live it is plausibly absent. `payment_provider` is actively trusted by the public storefront to route the card SDK (`liveStorefrontData.ts:73`).
**Exploit:** A malicious/confused owner runs `PATCH /rest/v1/organizations?id=eq.<own> {"square_payment_ready":true,"payment_provider":"square","square_merchant_id":"...","square_location_id":"..."}`. Combined with F9's `sandbox-direct` fallback, a self-activated `square_payment_ready` can misroute their card charges through the shared `SQUARE_ACCESS_TOKEN`. Even alone it forges processor identity/readiness or flips `payment_provider` with no real connection and breaks their own checkout.
**Fix (quick):** **Apply `guard_org_payment_columns` (`20260609050000`) to live NOW**, OR add an explicit `WITH CHECK` requiring `NEW = OLD` for these columns. Verify with a live test that an owner PATCH of `square_payment_ready` is a no-op. Do not rely on a migration documented as not-yet-applied.

### F8 — HIGH — Vault service-role-key drift silently kills the auto-decline + Square-refresh crons
**Location:** `migrations/20260609040000_respond_to_order_claim.sql:116-118,132` (`auto_decline_stale_orders`); `migrations/20260610020000_square_token_refresh_cron.sql:35-37,46` (`refresh_expiring_square_tokens`); `order-respond/index.ts:91` + `square-token-refresh/index.ts:38-39` (string-compare Bearer to env key). Also `migrations/20260422122034`.
**Risk:** The service-role key lives in two independent places: the edge-fn env `SUPABASE_SERVICE_ROLE_KEY` and a manual copy in `vault.decrypted_secrets`. The crons send the Vault copy as a Bearer; the edge fns authenticate by **exact string compare** to the env key. On rotation (MANDATORY after the admin-token leak), the env updates but the Vault copy does not auto-sync → every cron POST 401s → the job is silently dead. Both crons read Vault secrets and `RETURN` on NULL with no `RAISE`/log/alert, and discard the `net.http_post` result. This is the exact documented live failure (found by accident). For auto-decline this is a direct money bug: stale `awaiting_confirmation` orders never auto-decline, so manual-capture authorizations are never voided (Stripe/Square holds linger up to 7 days → chargeback liability). For Square-refresh: tokens age out → `CreatePayment` 401s for any merchant who doesn't transact within the inline-refresh window.
**Exploit:** Operator rotates the service-role key, forgets the Vault `service_role_key` secret. Within minutes both crons 401 silently. Card pre-auth holds accumulate unvoided; Square tokens expire and live charges start failing. First signal is a customer complaint, not an alert.
**Fix (medium):** Stop keeping a second copy of the key. Best: do the work in-DB — have `auto_decline_stale_orders` call `claim_order_for_response()` directly and enqueue the provider void without the self-HTTP hop. Or have the edge fns VERIFY the Bearer as a valid service-role JWT (signature) rather than `===`. **Exploit:** A developer or AI agent pastes the next service-role/Stripe/Square secret into a doc/commit/chat; nothing detects it; an attacker scraping the dev env, a leaked transcript, or git history harvests it and reaches the money path.
**Fix (medium):** (1) gitleaks/trufflehog as a pre-commit hook AND a required CI check that blocks on detection (covers `.env`, source, migrations, docs). (2) No-paste policy: secrets only via `supabase secrets set` / the dashboard / a vault. (3) Replace the `pk_live_...` placeholder with a clearly-fake value. (4) Execute a single rotation runbook + verify each. (5) Track per-secret last-rotated + scheduled rotation.

### F27 — MEDIUM — No monitoring/alerting on cron + payment edge-function failures
**Location:** `migrations/20260422122034:33` (auto_decline → order-respond), `20260610020000:44` (→ square-token-refresh); no heartbeat/cron-health table; no `net._http_response` reconciliation anywhere.
**Risk:** Both crons call `PERFORM net.http_post(...)` — fire-and-forget; the edge fn's 401/500/502 is discarded and never reconciled. Combined with F8, a rotated service-role key turns both crons into silent no-ops. The prior auto-decline death was found "by accident." With manual-capture payments, a silently-dead auto-decline means card pre-auth holds are never voided and no one is paged. This amplifies every other finding.
**Exploit:** Service-role key rotates → both crons 401 silently → days pass → holds accumulate and refresh tokens expire → live charges fail. First signal is a customer complaint.
**Fix (medium):** (1) Capture pg_net responses (reconcile `net._http_response` by request id) or have each cron-targeted fn write a heartbeat row (`last_success_at`, `last_status`) + alert when stale. (2) Route 4xx/5xx from payment + cron functions to an alerting channel (log drain → Sentry/Slack). (3) Mandatory post-rotation synthetic check that smoke-invokes each cron target and asserts 200.

### F28 — LOW — Stripe idempotency key omits amount; resume path doesn't re-check amount
**Location:** `stripe-payment-intent/index.ts:141` (`idempotencyKey: pi-${order.id}`), `:104-111` (resume returns existing `client_secret` with no amount check); contrast `square-payment/index.ts:337-340` (folds amount into key, H-3).
**Risk:** Currently inert (totals immutable post-creation), but a real defense-in-depth asymmetry: the Square path folds amount into its idempotency key and re-checks on resume; the Stripe path uses a fixed `pi-<order.id>` key and returns the existing `client_secret` without verifying `existing.amount === orders.total_amount`. If any future feature lets a total change post-PI-creation, Stripe replays the STALE amount and the fixed key blocks a corrected PI — a silent wrong-amount charge.
**Exploit:** Not exploitable today; becomes a wrong-amount charge the moment a later feature mutates `total_amount` post-PI-creation.
**Fix (quick):** Mirror the Square hardening: include `amountCents` in the key (`pi-${order.id}-${amountCents}`), and on resume verify the existing PI amount === current total before returning; if diverged, cancel the stale PI and create a fresh one.

### F29 — LOW — `square_payment_ready` has no DB invariant; `oauth.authorization.revoked` unhandled
**Location:** `square-payment/index.ts:162-176`; `square-webhook/index.ts:145` (revoke = "Future"); `migrations/20260609050000`.
**Risk:** The owner-self-set misroute is mitigated by the `20260609050000` guard (owner-immutable) and `square-payment` fails closed on null connection; `getFreshAccessToken` returns the merchant's OWN (possibly stale) token — never another's. So the misroute exploit is NOT reachable; the residual is a FAILED charge on a revoked/stale connection. Real gaps: (a) no DB invariant that `square_payment_ready=true` implies a `square_connections` row for `square_merchant_id`; (b) `square-webhook` doesn't handle `oauth.authorization.revoked`, so a merchant who revokes in Square keeps `square_payment_ready=true` and presents a broken-but-enabled card field.
**Exploit:** Merchant A revokes Square access; `square_payment_ready` stays true (no revoke webhook); customers see a card field whose authorize fails until manually fixed — degraded checkout, not misrouted funds.
**Fix (medium):** Handle `oauth.authorization.revoked`: flip `square_payment_ready=false` + delete/disable the connection. Add a DB invariant (trigger, or derive readiness from a join). Confirm the `20260609050000` guard is actually APPLIED to live (overlaps F7).

### F30 — LOW — `staff-pin-login verify_user` enumeration oracle; `get_member_org` over-exposes payment posture to staff
**Location:** `staff-pin-login/index.ts:65-85` (verify_user returns 200+display_name when username exists AND active, else 401; not attempt-counted — only the login action at `:120-128` counts); masking gap `migrations/20260610070000:107-125` (`get_member_org` staff branch nulls PII + Square ids + counters but NOT `payment_provider`/`charges_enabled`/`payouts_enabled`/`founding_merchant`/`settings`); staff reach via `src/services/api.ts:105`.
**Risk:** (1) `verify_user` returns 200+`display_name` for an existing ACTIVE staff username but generic 401 otherwise — a precise per-org username oracle (slugs are public), NOT attempt-counted, so an attacker freely maps valid (org, username) pairs + harvests display names, then focuses the rate-limited PIN brute force on confirmed accounts. PIN + lockout keep takeover impractical → impact is recon + display-name disclosure. (2) `get_member_org`'s staff branch leaves `payment_provider`/`charges_enabled`/`payouts_enabled`/`founding_merchant`/`settings` unmasked, so kitchen/service staff see the merchant's payment posture.
**Exploit:** Iterate `verify_user {org_slug, username}` — a 200+display_name confirms a real active login (no rate limit on this step). Separately, a kitchen staffer calling `get_member_org` reads payment posture.
**Fix (medium):** Make `verify_user` uniform regardless of existence (always 200 with a generic "enter your PIN"), and apply the login throttle to the verify_* steps. In `get_member_org`'s NOT-owner branch, null `payment_provider`/`charges_enabled`/`payouts_enabled`/`founding_merchant` and scrub `settings.payments`.

### F31 — LOW — Single global Square webhook signing key (defense-in-depth)
**Location:** `square-webhook/index.ts:84-110` (one `SQUARE_WEBHOOK_SIGNATURE_KEY`) + `:165-186`.
**Risk:** One platform-level signing key; a valid signature proves only that the configured app sent the event, not which merchant. `setOrderPayment` updates by `square_payment_id`/`reference_id` with no org scoping. Mitigated by the no-downgrade guard + unguessable ids, so hard to weaponize — but a money-state mutation has no org binding by design. (This is the structural sibling of F5; fixing F5 addresses both.)
**Exploit:** If an attacker obtains a valid `square_payment_id`/order UUID and a correctly-signed event (compromised app secret, or a future multi-app setup), they could push a non-terminal `payment_status` transition onto any org's order.
**Fix (medium):** Bind events to a merchant via the event `merchant_id` → `square_connections` before mutating; consider per-merchant webhook subscriptions/keys. (Covered by the F5 fix.)

### F32 — LOW — `get_public_storefront` returns the entire `settings` JSONB to anon
**Location:** `migrations/20260611000700` / `20260610070000` etc. — `settings` masked in 0 of them; consumed at `api.ts:215`, read at `RestaurantStorefront.tsx:626-628`, `PublishedStorefront.tsx:253-255`.

**Exploit:** Anon `rpc('get_public_storefront')` → `response.settings` exposes `payments.online_card_enabled` + `orders.confirmation_timeout_minutes`. ### F33 — LOW — Kitchen docket builder interpolates `order_source`/fulfillment unescaped (latent XSS)
**Location:** `src/lib/printDocket.ts:40,80` (banner from `order.order_source`), `:47,84` (service from `fulfillment_type`); written via `document.write` in `printKitchenDocket :92-97`; caller `Orders.tsx:393`.
**Risk:** `buildDocketHtml()` hand-builds HTML written into a popup (which inherits the authenticated merchant dashboard origin). `esc()` escapes only `& < >` and most fields go through it, but TWO derivations are interpolated RAW: `banner` (from `order.order_source`) and the non-dine-in `service`. Neither is attacker-controllable TODAY (no `orders.order_source` column exists; `fulfillment_type` is enum-cast in the RPC). Latent defense-in-depth gap.
**Exploit:** No live exploit. CLAUDE.md/memory record a planned `orders.order_source` column. If it lands as free-text from a client payload (like `table_number`/`notes`), printing a docket would render an attacker-supplied `online<img src=x onerror=...>` as live script in the print-popup (authenticated origin) → reads the Supabase session token from localStorage.
**Fix (quick):** Escape both sinks now: `<div class="banner">${esc(banner)}</div>`, and build `service` from `esc()`'d parts. Extend `esc()` to also encode `"` and `'` (mirroring `receipt.ts`).

### F34 — LOW — `storefront_config` validate trigger leaves section text props unbounded (single-layer XSS defense)
**Location:** `migrations/20260603010000_storefront_config.sql:47-125`; client parser `src/lib/storefrontConfig.ts:182-203`; render helpers `sections/types.ts:186-205`.
**Risk:** The trigger validates section ids, `_url` props (https-only), theme HSL/enums, hero copy length, and a 32KB cap — but does NOT cap length or restrict charset of arbitrary NON-`_url` scalar string props (headline/eyebrow/title/body). For hand-rolled PostgREST writes that bypass the client parser, the DB trigger is the only validation, allowing ~32KB of arbitrary unicode per prop. NOT XSS today (every section reads text via React-escaped `textProp()`/https-only `urlProp()`; zero `dangerouslySetInnerHTML`), but a single-layer (React-only) defense served to anon.
**Exploit:** No XSS today. (1) Storage/integrity: an org JWT holder POSTs ~32KB of arbitrary unicode/control chars into props text, served to every anon visitor. (2) Fragility: the moment any future section renders a text prop through an unsafe sink (a `dangerouslySetInnerHTML`, or an href/src/style built from a non-validated prop), the stored value becomes persistent XSS.
**Fix (medium):** Add belt-and-braces validation in `validate_storefront_config()`: recursively cap every string value's length (≤500) and reject ASCII control chars. Enforce the "React text nodes only, never `dangerouslySetInnerHTML`" rule with a CI lint/test.

### F35 — LOW — `courier-dispatch` forwards customer `shipping_address` verbatim + logs full upstream bodies
**Location:** `courier-dispatch/index.ts:107-146`.
**Risk:** No real SSRF (the URL is hardcoded `https://api.uber.com/...`). What IS real: `order.shipping_address` (written client-side at checkout) is forwarded UNVALIDATED/unbounded into the Uber manifest dropoff, and the error path logs the full upstream response body. Concrete harm is unvalidated-passthrough + verbose logging, not credential exposure.
**Exploit:** A customer places a delivery order with a crafted `shipping_address` (extra keys, oversized strings, control chars) that flows into the Uber manifest unvalidated; the error log captures full upstream bodies on failure.
**Fix (quick):** Validate + whitelist `shipping_address`/`business_address` fields (street/city/postcode/country only; length-capped, strip control chars) before building the body. Keep provider endpoints a hardcoded allowlist. Log status + request id only, never full upstream bodies.

### F36 — LOW — `courier-webhook` resolves order/secret from attacker `delivery_id` before signature verify (oracle)
**Location:** `courier-webhook/index.ts:38-85`.
**Risk:** The handler resolves the order by `payload.delivery_id` (attacker-controlled) and loads that org's webhook_secret BEFORE verifying the HMAC. Non-existent id → 200, missing secret → 503, bad signature → 401 — a response differential exposing which `delivery_id`s exist and which orgs configured a secret. Signature verification is correct and fail-closed (forging "delivered" still needs the per-org secret), so the oracle only aids targeting.
**Exploit:** Enumerate/replay `delivery_id`s; 200 vs 503 vs 401 reveals per-org existence + config state, narrowing targets for secret-guessing/replay.
**Fix (quick):** Return a uniform generic 2xx for unknown `delivery_id`s; don't branch HTTP status on internal existence/config. Prefer verifying a signature over the raw body BEFORE any DB lookup.

### F37 — LOW — `order-notify` lets kitchen-role staff send customer-facing notifications

### F38 — LOW — Anthropic upstream error bodies echoed to clients + logs
**Location:** `_shared/anthropic.ts:142-148,208-213`; surfaced via `json({error: e.message}, e.status)` in `ai-menu-copilot/index.ts:327-330`, `ai-inventory-assistant/index.ts:376-377`.
**Risk:** On a non-2xx from Anthropic, the client throws `AnthropicError(\`anthropic ${status}: ${txt.slice(0,300)}\`)` and the handlers return it to the client. This surfaces raw upstream text (model ids, request-shape, rate-limit specifics, fragments of the rejected prompt) to merchants + logs. The API key is never included → internal disclosure that aids prompt-injection crafting.
**Exploit:** A merchant triggers a malformed/oversized request and reads `{error:'anthropic 400: ...'}` containing prompt/model structure, then refines injection payloads.

### F39 — LOW — CSP allows `style-src 'unsafe-inline'` + wide `img-src https:`
**Location:** `public/_headers`; inline styles at `index.html:44,128-129`; tokens in localStorage at `client.ts:24-25,34-35`.
**Risk:** `script-src` is correctly free of `unsafe-inline`/`unsafe-eval` (the important half), but `style-src` permits `unsafe-inline`. Owner/customer/guest JWTs live in localStorage; a stolen owner JWT = full dashboard + payout surface. Inline-style injection enables CSS-exfiltration primitives. Defense-in-depth only (no current HTML-injection sink), compounded by `img-src https:` permitting exfil to any https host.
**Exploit:** No current sink. Latent: a future unescaped field or third-party widget enables injected `<style>` attribute-selector rules + `background:url()` to exfiltrate token/field values to any https origin, bypassing the strong script-src.
**Fix (medium):** Drop `unsafe-inline` from `style-src`: move the spinner keyframes + inline styles to an external hashed stylesheet (Tailwind output is already external) or add a style hash. Replace blanket `https:` in `img-src` with specific image hosts.

### F40 — LOW — CSP has no `report-to`/Reporting-Endpoints
**Location:** `public/_headers`.
**Risk:** The CSP blocks but never reports. On a project that already suffered a silent auth-key drift and pasted-secret incidents, the absence of CSP telemetry means an active injection probe, mis-loaded script, or payment-iframe origin change fails silently with zero alert — the same silent-failure class. Observability hardening.
**Exploit:** No direct compromise. An attacker probing for an injection sink triggers CSP blocks that generate no signal; the team learns only post-incident.
**Fix (quick):** Add `report-to csp-endpoint` to the CSP + a `Reporting-Endpoints: csp-endpoint="https://<sentry-collector>/..."` header (Sentry is already CSP-allowed). Introduce `style-src`/`img-src` tightening in report-only alongside enforce.

### F41 — LOW — Wildcard CORS (`*`) on payment/refund/staff-login/AI functions
**Location:** `square-payment`, `refund-order`, `stripe-payment-intent`, `order-respond/index.ts:5-9`, `staff-pin-login`, `_shared/anthropic.ts:303-306`.
**Risk:** Every function returns `Access-Control-Allow-Origin: '*'`. Auth is via bearer header (not cookies), so classic cookie-CSRF doesn't apply — not a direct takeover vector. The residual: `*` lets a script on ANY origin read responses cross-origin, removing cheap origin-pinning on the money path and broadening any leaked JWT's blast radius.
**Exploit:** A phishing page in a merchant's browser scripts fetch to `refund-order`/`square-payment` with a phished JWT; `*` lets it read the response to confirm success and enumerate, and run `staff-pin-login` brute-force from any origin.
**Fix (quick):** Replace `*` with an allowlist echo (`https://woahh.app`, `https://*.woahh.app`, localhost in dev) for the authenticated/payment/staff functions — validate `req.headers.get('origin')` and reflect it per-request. Even guest checkout runs on woahh.app origins.

### F42 — LOW — `Permissions-Policy payment=(self)` doesn't delegate to the Stripe iframe (latent wallet gap)
**Location:** `public/_headers`; Stripe `<PaymentElement>` `CardPayment.tsx:3,79`; Square card field `:40-41,187-192`.

If/when Apple/Google Pay buttons are enabled in `<PaymentElement>`, they run in the Stripe iframe and would be blocked. (Square's current integration is card-tokenize-only.) **Exploit:** No compromise. Wallet payments via Stripe `<PaymentElement>` silently fail/degrade if enabled while basic card tokenization still works (so a smoke test passes).
**Fix (quick):** Delegate minimally: `payment=(self "https://js.stripe.com")`. Add `web.squarecdn.com` only if/when a Square wallet path is introduced. Never `*`.

### F43 — LOW — External courier tracking link uses `rel="noreferrer"` only
**Location:** `OrderStatus.tsx:481-483,558-560`.
**Risk:** `rel="noreferrer"` already implies `noopener` on shipping browsers, so reverse-tabnabbing is mitigated today. The only untrusted target is the external courier URL. Relevant chiefly because the app is intended to be Capacitor-wrapped later, where the noreferrer→noopener implication is less guaranteed.
**Exploit:** No reproducible exploit on mainstream browsers. Theoretical: in a webview not implying noopener, a malicious `courier_tracking_url` could use `window.opener` to redirect the original (session-holding) tab to a phishing clone.
**Fix (quick):** Set `rel="noopener noreferrer"` explicitly on the `courier_tracking_url` anchors. Trivial.

## 6. What Is Already Good (the baseline)

The architecture is genuinely solid — these defenses are correct and should give the founder confidence in the foundation:

- **No PCI card storage.** Cards are tokenized at the edge — Stripe `<PaymentElement>` and Square Web Payments SDK `card.tokenize()`. woahh never sees or stores PANs; there is no cardholder-data surface to breach.
- **Server-side order-total validation (C1).** `create_order_with_inventory` recomputes the authoritative subtotal + promo and REJECTS any untrusted client total below the floor — the catastrophic "1¢ for an $80 order" undercharge is closed and live-verified. (The findings above are *adjacent* gaps — status, gating — but the amount floor itself holds.)
- **Per-org Square OAuth (the BLK-2 fix).** The design correctly moved off a single global token to per-org `square_connections`, with RLS deny-by-default and service-role-only access. The `sandbox-direct` affordance (F9) is a leftover hole *in* an otherwise-correct model, not a flawed model.
- **SECURITY DEFINER RPC masking.** Sensitive reads go through definer RPCs with deny-by-default RLS on the underlying tables — the right pattern. The weakness (F17) is the *denylist* style, not the use of definer RPCs; switching to allowlist projections strengthens an already-correct boundary.
- **Manual-capture payment flow.** Card-required online orders authorize-then-capture-on-owner-confirm (with auto-decline voiding stale holds). The model is correct; the gaps are operational (cron silently dying, status injection), not architectural.
- **Strong `script-src` CSP.** No `unsafe-inline`/`unsafe-eval` on scripts — the most important XSS control is in place. Webhook HMAC verification (Square, courier, Svix email) is timing-safe and fail-closed.
- **Tenant isolation discipline.** A single host→slug→org resolution path, `current_org_id()` kept free of `auth.users` joins (a documented hard-won lesson), and apex-vs-subdomain routing that keeps the isolation boundary intact. Most findings here are *within-tenant* integrity or *anon over-disclosure* — true cross-tenant data theft is rare and narrow.
- **Refund money-math + GMV/donation reconciliation** live in guarded RPCs (`record_order_refund`), and refunds route per-provider correctly. The gap (F16) is the *denormalized* mirror columns, not the authoritative ledger.

**Bottom line:** woahh is a well-architected platform with a containment problem (leaked keys) and a regression problem (the storefront mask). Neither requires re-architecting. Rotate the keys today, ship the handful of quick-win migrations this week, and the effective posture jumps from C+ to B+/A- quickly.
