## (2) WHERE THE FOUNDER'S ASKS ARE RISKY OR WRONG — and the better path

1. **#11 — one checkbox for Terms + marketing is a Spam Act violation. Do not ship it.** Bundling marketing consent into a mandatory Terms tick is exactly the "not freely given" pattern the Spam Act 2003 prohibits, and *you* (the platform) are on the hook, not the merchant. The current code is already correct: T&C required, marketing **optional + pre-unchecked** (`RestaurantStorefront.tsx:267-268`, opt-ins at `:1563`/`:1600`). The requested "improvement" would *break* a compliant flow.
   **Better path:** keep one **required** Terms checkbox; keep marketing as a **separate, optional, unchecked** line directly beneath it. You can make it visually tighter (one card, two ticks) without legally bundling.
   **Note:** the punch-list cites `docs/legal/legalities.md §6` — **that file does not exist** (`find docs -iname "*legal*"` returns nothing matching). Whoever wrote the note was citing a doc that isn't there.

2. **#1 — hard ABN gate on ALL payment config is launch friction with a single in-person merchant.** ABR/ABN lookups have lag and edge cases (sole traders, pending registrations, trust structures). If merchant #1 is sitting next to you and the gate blocks them from configuring *pay-at-venue* (zero card/AFSL/charity-handling risk), you've manufactured a launch-day blocker for no compliance benefit. The list flags this is held tonight — good.
   **Better path:** when you build it, gate **online card acceptance only** on `abn_verified` (the AFSL/funds-handling surface). Pay-at-venue config must never be ABN-gated.

3. **Gating ALL payment instead of just online-card (#1 + #13 collide).** #13 wants a merchant toggle for pay-at-venue vs online; #1 wants to hard-gate "payment acceptance (online card + pay-at-venue config)." Pay-at-venue is cash/EFTPOS the merchant already legally takes — woahh touches no funds. Gating it adds zero safety and real friction.
   **Better path:** gate online-card on ABN; never gate pay-at-venue. The plumbing already supports the split — `online_card_enabled` is the single online-card switch (`src/services/settings.ts:66`, default false `:197`).

4. **The owner-preview UX trap is real and worse than it looks.** `useIsOwnerPreviewing` (`src/hooks/useIsOwnerPreviewing.ts`) suppresses the online-card path when the *logged-in owner* views their own shop (`PublishedStorefront.tsx:405` `payOnline = onlineCardEnabled && !isOwnerPreviewing && !dineIn`). The trap: **the merchant testing their own checkout will NEVER see the card form**, will conclude "online payments are broken," and call you. Commit `d0e8a69` adds an owner-preview *notice* — good, but a banner is a weak fix.
   **Better path:** give the dashboard an explicit "Preview as customer (test mode)" button that opens the storefront with card *shown* but routed to Square **sandbox**, so the merchant runs a real end-to-end test card with no live charge. Kills both the confusion AND the "I never tested my own checkout before going live" risk.

5. **#14 is mislabeled. The charity/commission flow doesn't exist server-side — there is nothing to "fix."** Grepped every edge function and migration: nothing inserts `gmv_mandatory` rows into `donation_ledger` on order/capture. The only `donation_ledger` inserts are voluntary/one-time (owner-initiated, `20260423085555...sql:88`) and a refund reference (`20260610050000_order_refunds.sql`). Stripe `application_fee_amount` is **hardcoded 0** for founding merchants (`stripe-payment-intent/index.ts:121`); Square `app_fee_money` only set when positive (`square-payment/index.ts:352`). So for your founding merchant: **0% is taken, 0 charity rows are written, /impact shows nothing from real orders.** That is *consistent* with the founding-merchant 0%-commission deal — but #14 isn't "fix the flow," it's **"decide whether founding merchant #1 generates ANY charity ledger entries, and if so, build the write"** (a trigger on capture, or a write inside `order-respond`). Don't let it read as a bug-fix; it's a product decision + net-new server write.

## (3) WHAT'S MISSING — what merchant #1 hits on day 1 that's NOT on the list

1. **No "view/preview as customer" + the owner-preview card-hiding = guaranteed support call.** (See 2.4.) The #1 day-1 confusion. Not on the list at all.
2. **The demo-flag leak is a live footgun.** `isDemoMode()` is a single localStorage key `lumen.demo.active` (`src/lib/demo.ts:25, 2007`). If the founder (or merchant) ever visited `/demo`, the flag persists, and `services/api.ts` silently routes **every** product/order write to the in-memory store — the code even logs "product is going to the in-memory demoStore, NOT the real database" (`api.ts:278`). A merchant who played with the demo then logs into their real account can create products that vanish and never reach Square. **Add a loud demo banner in the real dashboard + auto-clear the flag on real merchant login.** A correctness landmine, not polish.
3. **Receipt/docket print path is partial.** Email receipt for online orders done (`afd0070`), "ONLINE ORDER" docket exists (`e39b6a2`), but **no KDS print button** and **no in-person/PDF receipt** — backlog `[4.3]`. A restaurant with a kitchen printer expects to print the ticket. Comes up hour one.
4. **Refunds UX exists but is unverified against Square.** `refund-order` + the Orders.tsx dialog (`:218-258`) are built, but refund→donation-ledger reconciliation references the flow (#14) that isn't writing on the way *in*. If you never wrote the charity row on capture, refunding it is incoherent. **Verify the full Square money round-trip (capture → refund) in sandbox before launch** — not on the list.
5. **Zero monitoring / error alerting.** No Sentry, no edge-function error alerting, no "order failed to capture" notification. With one merchant, you are the monitoring. Realistic gap: a Square capture fails silently in `order-respond`, the merchant just doesn't get paid, and nobody knows. **Add at minimum a structured-log alert (even a webhook to your phone) on payment capture/refund failure.** Cheap; it's the difference between "I called the merchant" and "the merchant noticed three days later."
6. **Schema-drift debt is actively biting.** Your last ~6 commits are drift hotfixes (`6433356`, `a75da17`, plus three repairing `get_public_storefront`/`get_order_by_id`/`get_member_org`). Repo migrations ≠ live DB. **Reconcile repo migrations against the live `pmnyhbhtkcfoozkinieo` schema once and regenerate `types.ts` before merchant #1**, or every deploy is a coin-flip and you'll be hotfixing RPCs during the dinner rush.
7. **Key rotation is overdue and on no active list.** Your own memory says `sbp_`/`sb_secret_`, ClickSend, Anthropic, Resend, GitHub PATs were pasted in chat and need rotating (backlog `[ops]`). You're about to handle a real merchant's payment connection and a real customer's PII. **Rotate before live, not after.** Founder-does-it-in-30-min, but must precede go-live.
8. **No prep-time/accuracy at all on the bespoke path (the actual first merchant).** Your first published merchant uses the bespoke ThemeShell path. Confirm `screens/Checkout.tsx` shows a real ETA — it currently has no `etaMinutes` equivalent. #10 must cover the bespoke path or your real first merchant ships with zero pickup-time display.

## (4) CONCRETE IMPROVEMENTS (3–5, build-ready)

1. **Reframe the launch around a single "Go-live readiness" gate, not 17 polish items.** One dashboard screen running real checks: ABN verified? → Square connected (test charge succeeded in sandbox)? → ≥1 product? → hours set? → prep-time set? → storefront published? Each row links to the fix. **Absorbs #1/#2/#5**, makes #7's publish-gate the explicit "go-live" action, and gives you a one-glance "is this merchant safe to turn on" view. Highest-ROI thing you could build, and it's not on the list.
2. **Ship a "test-charge" sandbox preview button (solves 2.4 + 3.1 + 3.4 at once).** One dashboard button: "Run a test order." Opens the storefront with card shown, routed to Square **sandbox**, places a real test order end-to-end (capture + receipt email + docket). The merchant *sees their own checkout work* before going live, you verify the full money path, and you kill the owner-preview confusion. De-risks the launch more than any three list items.
3. **Make the publish action the real go-live switch and gate the DEFAULT path too.** Today "published" only means anything on the bespoke template path. Add an org-level `storefront_live` (or reuse `marketplace_visible` consistently) checked in `orgApi.getBySlug`/`Shop.tsx`, so an org that hasn't pressed "Go live" returns a "coming soon" screen on *both* render paths. Makes #7 honest and gives #5's checklist a meaningful terminal action. Today a merchant is live before they intend to be.
4. **Do #10 (prep time) as a real, simple merchant input — don't over-engineer it.** No dynamic kitchen-load ML. Add `settings.fulfillment.prep_time_minutes` (pickup) + a delivery offset, show `prep_time + busyBufferMinutes`, and let the merchant bump the busy-buffer with one tap on KDS during a rush (`busyBufferMinutes` already plumbed). Accurate-enough beats fancy. Wire into *both* `RestaurantStorefront.tsx:395` and `screens/Checkout.tsx`.
5. **Cut the list in half and sequence it** (see section 1). **Explicitly do NOT do #11 as written, and do NOT flip guest-secondary #15 for launch.**

## (5) DECISION BRIEF — merchant public web address (TODO #8 / domain strategy)

**Scope:** item #8 (publish→activate subdomain) is HELD tonight — this is the input for that decision.

### Ground truth (what is actually in the code today)
- **Live render path is PATH-BASED.** A storefront is reachable at `woahh.app/shop/<slug>` → `src/pages/Shop.tsx:37` (`slug = forcedSlug ?? params.slug`) → `orgApi.getBySlug(slug)` → `get_public_storefront(p_slug)` RPC. Only path that works in prod. Dashboard "view storefront" confirms it: `src/pages/dashboard/StorefrontTemplates.tsx:373-374` builds `apexUrl('/shop/' + org.subdomain_slug)`.
- **Subdomain resolution is CODED but not live.** `src/lib/tenant.ts` fully parses `<slug>.woahh.app` (`resolveTenant`, `RESERVED_HOSTS` `:24`, `SLUG_RE` `:38`, fail-safe-to-apex), wired into `src/App.tsx:95,127` and `Shop.tsx:76-94` (per-merchant PWA manifest). Missing = purely human/infra: wildcard `*.woahh.app` DNS + TLS + Cloudflare Pages custom domain.
- **"Publish" today controls the RENDER PATH, not the domain.** `StorefrontTemplates.tsx:332` writes `storefront_config.is_published = true`; `Shop.tsx:117-132` then renders bespoke `PublishedStorefront` instead of default `RestaurantStorefront`. No domain activation wired to publish — so #8 is genuinely net-new behaviour.
- **Custom domains do NOT exist in code.** Grep for `get_org_by_host`, `custom_domain`, `set_subdomain_slug`, `reserved_slugs`, `slug_aliases` across `src/`+`supabase/` returns nothing. Described only in `docs/DOMAINS_ROADMAP.md`. Option C is greenfield.
- **The slug-change security gap (R1) is real and unbuilt.** `tenant.ts` has the reserved list, but there's no `reserved_slugs` table, no `set_subdomain_slug` RPC, no `guard_subdomain_slug` trigger. A merchant could `UPDATE organizations SET subdomain_slug='admin'` via REST today. Bites option B.

### Option A — Path-based `woahh.app/shop/<slug>` (today)
- **SEO/branding:** weakest brand signal — URL/address bar/share previews read "woahh.app," not the merchant; link equity accrues to apex. Functionally fine (canonical at `/shop/:slug`, indexable). OK for marketplace-discovery framing, weak for "this is *your* online shop."
- **Technical work:** **zero** — live now. Only #8-adjacent work is the visibility gate (TODO #7: unpublished must 404).
- **Cost/complexity:** none. No new DNS/TLS/per-host infra.
- **Isolation/security:** strongest — single origin, single localStorage namespace, one CSP, the already-audited `get_public_storefront` RPC boundary. Slug is a query param, never a host/authz input — no slug-squatting surface.
- **#8:** collapses to "publish makes `/shop/<slug>` publicly reachable" (= #7's gate). The fallback if B's infra slips.

### Option B — Per-merchant subdomain `<slug>.woahh.app` (scaffolded plan)
- **SEO/branding:** strong. `tacojoint.woahh.app` reads as the merchant's own site; address bar, share cards, installed-PWA name carry the merchant brand (`pwaManifest.ts` swaps name/icon/theme, wired `Shop.tsx:88-94`). Each subdomain is its own indexable origin; plan canonicalises `/shop/:slug` → `tenantUrl(slug)` so the two don't compete. The "every merchant has their own app" positioning.
- **Technical work (build-ready):**
  1. **Human/infra (blocks everything):** wildcard CNAME `*` → `<project>.pages.dev` (proxied), add `*.woahh.app` as a Pages custom domain, order a Cloudflare **Advanced Certificate** for `woahh.app + *.woahh.app` (Universal SSL does NOT cover wildcards), SSL Full (strict). Add `https://*.woahh.app/**` to Supabase Auth redirect allow-list.
  2. **R1 slug-mutation hardening (REQUIRED before go-live):** migration — `reserved_slugs` table (seeded from `RESERVED_HOSTS`, `tenant.ts:24-35`), `set_subdomain_slug(p_slug)` SECURITY DEFINER RPC (lowercase + `SLUG_RE` validate + reserved-check + uniqueness + ≤1 change/24h), `guard_subdomain_slug()` BEFORE-UPDATE trigger blocking out-of-band changes. Without this, a wildcard zone lets a merchant claim `admin.woahh.app`.
  3. **#8 publish→activate wiring (the founder item):** subdomain resolves via `get_public_storefront` independent of `is_published`, so #8 needs an explicit gate. Cleanest: gate the subdomain root on a published/`storefront_live` flag, and surface the live `tenantUrl(slug)` link in `StorefrontTemplates.tsx` only after publish (replacing the `/shop/<slug>` href at `:373-374`).
  4. **Apex guard (partly done):** `ApexOnly` already wired in `App.tsx` so `/business`+`/eat` bounce to apex from a tenant host — keeps merchant auth single-origin.
- **Cost/complexity:** low ongoing — one Advanced Certificate (~$10/mo, covers all merchants under the wildcard) + the wildcard CNAME. No per-merchant DNS/cert. One org row = one subdomain. This is the cost-efficiency edge over Shopify/Bopple.
- **Isolation/security:** strong-but-adds-surface. Per-origin localStorage keeps customer sessions tenant-scoped (isolation-positive). New risks: R1 slug-squat (above); R2 must NEVER set a `Domain=.woahh.app` cookie (would bleed across tenants — auth is localStorage so it holds; add a runtime guard); R6 slug change orphaning installed PWAs/QRs (needs `slug_aliases` 301). Data-plane trust boundary unchanged — host header never authz input; every read flows through the same PII-nulling RPC.
- **TLS/DNS:** wildcard covers ONE label only (`*.woahh.app` ✅, `*.*.woahh.app` ✗) — `SLUG_RE` enforces single-label slugs, by design.
- **#8:** native home — extend publish so it (a) flips a live flag the subdomain root honours and (b) swaps the dashboard link to `tenantUrl(slug)`. Until publish, `<slug>.woahh.app` shows "coming soon"/404, satisfying #7 + #8 together.

### Option C — Merchant's own custom domain `taco-joint.com.au` (paid add-on)
- **SEO/branding:** strongest — full brand ownership, all link equity on the merchant's domain. The Shopify/Squarespace premium tier.
- **Technical work:** greenfield. Build seam in `DOMAINS_ROADMAP.md:29-34`: (1) `organizations.custom_domain` + unique index; (2) anon `get_org_by_host(host)` SECURITY DEFINER RPC mirroring `get_public_storefront`; (3) extend `resolveTenant` — a non-apex/non-`*.woahh.app` host is unpredictable, so it needs a runtime DB lookup (new async path; `resolveTenant` is currently pure/sync — the one architectural change C forces); (4) per-domain Supabase Auth allow-list; (5) a "connect a domain" wizard. At scale needs **Cloudflare for SaaS** (Custom Hostnames API) for per-domain auto-certs. Two flavours: Tier 2 "buy through us" (Registrar API + auto-renew tied to billing — expiry = dark site, the #1 risk) vs Tier 3 BYOD.
- **Cost/complexity:** highest — per-hostname cost, ownership verification, apex CNAME-flattening/ALIAS handling, auto-renew billing reconciliation, abuse/trademark T&Cs, portability obligations. Real per-domain ops burden.
- **Isolation/security:** comparable to B once `get_org_by_host` mirrors the existing RPC, plus a per-domain TLS pipeline, domain-verification spoofing surface, and the async-resolution change to `tenant.ts` (host→org DB lookup on cold load — must fail-safe like the slug path).
- **TLS/DNS:** per-domain certs; apex domains need ALIAS/flattening or www+redirect; ownership TXT/CNAME verification before issuance.
- **#8:** out of scope — a custom domain is activated by a connect-wizard + DNS verification, not the publish toggle.

### RECOMMENDATION
**Ship Option B (`<slug>.woahh.app`) as the publish target, with Option A as the zero-risk fallback if the wildcard infra isn't ready.**

- B's code is **already written and wired** (`tenant.ts`, `App.tsx` root split, `Shop.tsx` forcedSlug + manifest, `ApexOnly`). Remaining = one human infra step (wildcard DNS+TLS+Pages domain) + two code deliverables (R1 slug-hardening migration; publish→live-flag + dashboard-link swap). Small, bounded, large branding win vs Bopple.
- B is the natural home for #8 — "publish" becomes the moment the branded subdomain goes live, folding in #7 (unpublished = not publicly reachable) cleanly.
- **Hard prerequisite before B goes live:** the R1 slug-mutation guard (`reserved_slugs` + `set_subdomain_slug` RPC + `guard_subdomain_slug` trigger). With a wildcard zone, an unguarded `subdomain_slug` UPDATE is a reserved-host hijack. Gating security item, not optional.
- **Fallback:** if you won't provision wildcard TLS before merchant #1, ship #8 against Option A ("publish makes `/shop/<slug>` reachable + the #7 visibility gate") and cut over the *same publish button* to the subdomain later — the dashboard link at `StorefrontTemplates.tsx:373-374` is the single line that changes from `/shop/<slug>` to `tenantUrl(slug)`.
- **Option C is post-launch, paid, deferred** — greenfield, real per-domain ops/cost, forces the only structural change to `tenant.ts` (sync→async). Not needed for the first merchant; should not block launch.
