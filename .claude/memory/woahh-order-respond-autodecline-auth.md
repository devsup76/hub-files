Live debugging, 2026-06-12. The auto-decline cron (`auto_decline_stale_orders()`) was 401'ing every minute (6798 failures; 21 orders stranded in `awaiting_confirmation`, Square holds not released). The "#19 Vault re-sync" instinct was a RED HERRING and cost a long rabbit hole.

**Real root cause:** the overnight hardening (commit `a60131f`) made `order-respond` action `auto_decline` require `bearer === SUPABASE_SERVICE_ROLE_KEY` *byte-exact*. The cron reads its bearer from Vault `service_role_key`; that copy was a **valid** service-role JWT (`role: service_role`, correct ref, issued at project creation 2026-05-30, ends `…Bo-8Ew`) but **NOT byte-identical** to the runtime-injected `SUPABASE_SERVICE_ROLE_KEY`. A Supabase project can legitimately hold **more than one valid service-role token** (same JWT secret, different copy/iat) — `square-payment` authenticated fine with the injected one while order-respond rejected the Vault one. The dashboard `service_role` and the Vault matched each other (`…Bo-8Ew`) but neither equalled the injected key, so **no key edit could ever fix it** — the dashboard doesn't expose the runtime's copy.

**Fix (committed `200619e`, deployed v40, verify_jwt stays TRUE):** accept the exact injected key OR any token whose decoded `role` claim is `service_role`. Secure because `verify_jwt=true` makes the gateway validate the signature before the code runs — we only confirm the service_role claim. Live-verified 401→200, all 21 stranded orders auto-declined, payment paths unchanged.

**How to apply / diagnose next time:**
- For a cron→edge-function 401, read `net._http_response` (status_code + content) — it shows the ACTUAL HTTP status the pg_net call got. `cron.job_run_details` says "succeeded" even when the inner http leg 401s (pg_net is async), so it's misleading.
- Distinguish gateway-401 vs code-401 by the body shape: gateway = `{"code":"UNAUTHORIZED_NO_AUTH_HEADER",...}`; order-respond code = `{"error":"Unauthorized"}`.
- Don't assume injected `SUPABASE_SERVICE_ROLE_KEY` == the dashboard/Vault service_role key. They can differ. `service_role` keys do NOT rotate themselves.
- Auto-mode classifier (correctly) blocks: `api-keys?reveal=true`, deploying a new public fn that echoes key fingerprints, and prod payment-fn deploys without explicit user authorization. Get explicit approval (AskUserQuestion) before deploying payment-adjacent auth changes.

**INCIDENT + lesson (2026-06-12, same day):** the F46 defect-fix (migration `20260612140000`, applied via the bundle) broke ALL order placement — `clamp_order_text` (a BEFORE INSERT trigger on `orders`) built its control-char strip regex with `chr(0)..chr(8)`. `chr(0)` is a literal NUL byte → Postgres "null character not permitted" (SQLSTATE 54000) on every insert → customers saw the generic "could not place order". ROOT LESSON: the defect-fix agent wrongly assumed Postgres ARE doesn't support `\xNN` inside bracket classes and "fixed" a non-bug with `chr()` — I had ALREADY verified `[\x00-\x08]` works on PG17 yet shipped the chr() version anyway. NEVER use `chr(0)` in a stored regex; the `[\xNN]` literal form is correct (the pattern string holds the text `\x00`, not a NUL). Hotfixed live + repo (`94f14ac`) back to `[\xNN]`, kept the jsonb_set shipping fix. DIAGNOSIS TECHNIQUE that worked: reproduce the order RPC via Management API `database/query` inside a `do $$ ... raise exception '%', v_result; end $$` block (rolls back any insert) — and `set_config('request.jwt.claims', json_build_object('sub', <customers.user_id>, 'role','authenticated')::text, true)` to get past the RPC's `auth.uid()`/"Sign in is required" gate.
